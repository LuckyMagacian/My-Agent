## name: software-dev-workflow  
description: 指导软件开发与变更的标准化工作流程。包含三种模式：完整需求开发流程（需求→设计→实现）、功能变更流程（澄清→计划→确认→实施）和问题修复流程（分析→方案→影响评估→确认→修复）。当用户发起新功能开发、需求分析、架构设计、功能修改、Bug修复或任何代码变更任务时自动应用。
# 软件开发与变更流程
根据任务性质自动匹配工作流：

| 工作流 | 适用场景 | 特征 |
| --- | --- | --- |
| **完整需求开发** | 从零开始或涉及完整功能模块 | 需要需求→设计→实现全流程 |
| **功能变更** | 在已有项目中进行小范围改动 | 局部功能修改、样式调整 |
| **问题修复** | 修复已有功能的异常行为 | Bug修复、逻辑错误修正、状态不一致修复 |


**判断原则**：若改动涉及新增限界上下文、聚合根或跨模块架构变更，走完整需求流程；若是修复已有功能的异常行为，走问题修复流程；否则走功能变更流程。

---

## 一、完整需求开发流程
### 阶段 1：需求开发
**目标**：与用户充分沟通，形成结构化、可追溯的需求文档。

1. **需求收集**
    - 通过提问 + 提供选项的方式与用户沟通，避免遗漏关键需求
    - 对模糊需求主动追问，给出可选方案并标注推荐项
    - 分析项目中已有的相关模块和可复用组件
2. **需求分类与结构化**
    - **功能性需求**：核心业务功能、用户交互流程
    - **非功能性需求**：性能、安全、可用性、可维护性、可观测性
    - **接口需求**：IPC接口、事件接口、外部系统集成
    - **数据需求**：数据模型、存储方案、数据生命周期
    - 每条需求须包含：唯一标识（如 `REQ-001`）、优先级（MoSCoW）、验收标准
3. **文档输出** → `工作目录/docs/需求说明书.md`（增量更新并标注变更版本）

**质量检查**：每条需求有唯一标识和优先级 | 非功能性需求已覆盖 | 用户已确认

---

### 阶段 2：概要方案设计
**目标**：基于DDD理念，设计系统架构方案。

1. **领域分析**：识别限界上下文、定义聚合根/实体/值对象、建立统一语言术语表、绘制上下文映射图
2. **架构设计**：DDD四层分层（Interface → Application → Domain → Infrastructure）、组件划分与职责、交互模式、数据架构
3. **技术选型**：列出语言/框架/中间件清单，说明选型理由和权衡取舍
4. **文档输出** → `工作目录/docs/设计说明书.md` 或 `工作目录/docs/design/` 下多文件

**质量检查**：聚合边界清晰 | 分层职责明确 | 用户已确认

---

### 阶段 3：详细方案设计
**目标**：将概要设计细化到模块/类级别，可直接指导编码。

1. **接口设计**：API契约、DTO定义、IPC Command定义
2. **数据库设计**：ER图、表结构、索引策略、迁移脚本
3. **核心逻辑设计**：算法流程、状态机、领域事件、错误处理策略
4. **前端架构设计**：组件树、状态管理、路由导航
5. **文档输出** → `工作目录/docs/design/{序号}-{模块名称}.md`

**质量检查**：接口入参/出参/异常明确 | 表结构含字段类型和约束 | 关键流程有时序图 | 用户已确认

---

### 阶段 4：实施编码
**实施顺序（严格遵循）**：

1. **先后端，后前端**
2. **后端内部**：Domain层 → Infrastructure层 → Application层 → Interface层
3. **先静态，后动态**（先定义结构/类型，再实现逻辑）
4. **先结构，后内容，再样式**（前端UI开发顺序）

**编码规范**：

+ Application层只做编排，不写业务规则
+ Domain层不依赖Infrastructure层（依赖倒置）
+ Infrastructure层实现Repository接口
+ 每完成一个模块须验证编译通过后再继续

## 二、功能变更工作流程

**遵循原则**：
1. **最小改动原则**：只修改与变更需求直接相关的代码，不做无关重构或"顺手优化"
2. **向后兼容原则**：变更不得破坏已有功能的正常行为，接口改动须保持向后兼容
3. **架构一致性原则**：新增/修改代码须遵循现有架构风格、分层规范和命名约定
4. **可回退原则**：变更方案须考虑回退策略，确保在出现问题时能快速还原
5. **渐进式变更原则**：优先拆分为多个小步骤逐步交付，避免一次性大规模改动

### 步骤 1：需求澄清
+ 通过提问 + 提供选项与用户沟通
+ 明确变更范围、影响面、预期行为
+ 分析相关代码和可复用组件，识别潜在风险

### 步骤 2：制定计划
+ 生成具体改动计划，列出需修改的文件和改动要点
+ 保存到 `工作目录/docs/modify/{改动名称}.md`

### 步骤 3：用户确认
+ **必须**要求用户查看改动计划
+ 用户确认后方可执行（如"确认"、"开始实施"）
+ **未收到确认指令前，不得开始实施变更**

### 步骤 4：实施变更
+ 严格按确认后的计划执行
+ 每完成一项改动验证编译/运行是否正常
+ 发现计划外问题及时与用户沟通调整

## 三、问题修复(bugfix)工作流程

**遵循原则**：
1. **先理解后修复原则**：修复前必须充分理解现有代码的设计意图和上下文，避免"修一个Bug引入三个新Bug"
2. **根因修复原则**：必须定位并修复根本原因，禁止仅做表面症状的 workaround
3. **最小侵入原则**：优先选择影响范围最小、改动量最少的修复方案
4. **防御性加固原则**：修复时应评估并增加防御性代码（参数校验、边界检查、异常兜底），防止同类问题再次发生
5. **回归验证原则**：修复后必须对影响范围内的相关功能进行验证，确保未引入新问题

### 步骤 1：问题分析
+ **复现问题**：明确问题的触发条件、现象和复现步骤
+ **定位根因**：通过日志、调试、代码追踪等手段，定位到引发问题的根本原因（而非表面症状）
+ **归类问题**：判断问题类型（逻辑错误 / 状态不一致 / 数据异常 / 接口契约不匹配 / 边界条件遗漏 / 并发竞态）
+ **记录发现**：将根因分析结论写入 `工作目录/docs/modify/{问题名称}.md`，包含：问题描述、复现步骤、根因分析

### 步骤 2：制定修复方案
+ **方案设计**：针对根因提出修复方案，优先选择最小侵入性的改动
+ **备选方案**：若存在多种修复路径，列出各方案的优劣对比，标注推荐项
+ **防御性加固**：评估是否需要在相关位置增加防御性代码（参数校验、边界检查、异常兜底），防止同类问题再次发生
+ **方案文档化**：将修复方案追加到 `工作目录/docs/modify/{问题名称}.md`

### 步骤 3：影响范围分析
+ **直接受影响**：列出修复直接修改的文件、函数、数据结构
+ **间接受影响**：分析调用链上下游、依赖该模块的其他功能、共享状态或数据的组件
+ **回归风险评估**：评估修复是否可能引入新问题，标注需要重点回归验证的功能点
+ **影响范围文档化**：将影响分析追加到 `工作目录/docs/modify/{问题名称}.md`

### 步骤 4：用户确认
+ **必须**要求用户查看修复方案及影响分析
+ 用户确认后方可执行（如"确认"、"开始修复"）
+ **未收到确认指令前，不得开始实施修复**

### 步骤 5：实施修复
+ 严格按确认后的方案执行，不做方案外的额外改动
+ 每完成一项改动验证编译/运行是否正常
+ 修复完成后，针对影响范围进行回归验证，确保未引入新问题
+ 将修复结果和验证情况追加到 `工作目录/docs/modify/{问题名称}.md`

---

## 四、通用约束
1. **文档先行**：设计和需求必须先文档化，经用户确认后再编码
2. **增量演进**：每步完成后验证，避免大批量改动后才发现错误
3. **变更可追溯**：所有文档变更须标注变更内容和原因
4. **错误处理**：任何阶段的实现都必须包含错误处理设计，不允许忽略异常场景
