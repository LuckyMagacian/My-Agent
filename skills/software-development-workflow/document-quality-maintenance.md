# 文档质量保障机制

## 问题背景

长期迭代后文档常见问题：
1. **过时失效**：代码已修改，文档未同步更新
2. **碎片化**：同一主题有多个版本，不知哪个最新
3. **质量退化**：新文档不遵循规范，风格不一致
4. **冗余堆积**：历史文档未归档，难以查找

---

## 一、文档生命周期管理

### 1. 状态标记

每个文档必须包含状态元数据：

```markdown
---
status: draft | active | completed | deprecated | archived
created: YYYY-MM-DD
updated: YYYY-MM-DD
version: 1.0
related_to: [相关需求/设计文档]
---
```

**状态定义**：

| 状态 | 含义 | 可见性 | 适用场景 |
|------|------|--------|---------|
| `draft` | 草稿，未完成 | 仅作者可见 | 文档编写中 |
| `active` | 活跃，当前有效 | 所有人可见 | 开发进行中 |
| `completed` | 已完成，不再变更 | 所有人可见 | 功能已上线/修复已合并 |
| `deprecated` | 已废弃，不再维护 | 标记删除时间 | **明确要求废弃** |
| `archived` | 已归档，仅作历史记录 | 移至 archive/ | 所有状态30天后 |

### 2. 状态转换规则

```
draft → active      : 用户确认后
active → completed  : 功能已上线/修复已合并/实施完成
draft → deprecated  : 用户明确要求废弃（未完成文档）
active → deprecated : 用户明确要求废弃（进行中文档）

所有状态 → archived : 30天后自动归档
```

**核心原则**：
- **completed 不进入 deprecated**：已完成表示工作落地，只能直接归档
- **deprecated 仅用于未完成文档**：draft 或 active 状态可废弃
- **新版本不影响旧版本**：创建新版本文档时，旧版本保持原状态
- **统一归档周期**：所有状态都是 30 天后进入 archived

### 3. 状态转换详解

#### draft → active（草稿生效）

**触发时机**：
1. **用户明确确认**：用户说"确认"、"可以"、"没问题"
2. **文档完整性检查通过**：
   - 无 TBD/TODO/待补充 占位符
   - 必需章节全部完成
   - 关联文档已建立

**执行动作**：
- 更新 `status: active`
- 更新 `updated: 当前日期`
- 记录到 INDEX.md 的 Active 区域

**示例场景**：
```
AI: 需求说明书已完成，请确认内容是否正确？
用户: 确认，继续下一步
→ AI: 更新状态为 active，开始设计阶段
```

---

#### active → completed（活跃完成）

**触发时机**：

| 文档类型 | 完成条件 |
|---------|---------|
| 需求说明书 | 功能已上线且验收通过 |
| 设计说明书 | 代码实现完成且符合设计 |
| 详细设计 | 模块开发完成且测试通过 |
| 变更计划 | 变更已实施并验证通过 |
| 问题修复 | Bug 已修复且代码已合并 |
| 分析报告 | 分析完成且已归档 |

**执行动作**：
- 更新 `status: completed`
- 添加 `completed_date: 当前日期`
- 添加 `completion_evidence: 证据链接`（如 commit/PR/上线记录）
- 移至 INDEX.md 的 Completed 区域

**completed 的作用**：
1. **明确交付**：表示文档对应的工作已落地完成
2. **不再变更**：后续修改应走变更流程或新版本
3. **可追溯**：保留完成证据和交付记录
4. **减少维护负担**：不再参与定期审计检查

**示例场景**：
```
场景 1：功能上线
用户: 用户登录优化功能已上线，验收通过
→ AI: 将需求/设计/详细设计文档状态更新为 completed
→ 记录上线日期和版本号

场景 2：Bug 修复
用户: Bug 已修复，PR 已合并
→ AI: 将问题分析文档状态更新为 completed
→ 记录 PR 链接和合并日期
```

---

#### draft/active → deprecated（明确废弃）

**触发时机**：
- **用户明确要求**：用户说"废弃这个文档"、"不再使用"
- **需求取消**：用户说"这个需求取消了"

**重要**：
- 新版本发布**不会**自动废弃旧版本，旧版本保持原状态直到自然归档
- **completed 状态不能进入 deprecated**，已完成的工作只能直接归档

**执行动作**：
- 更新 `status: deprecated`
- 添加 `deprecated_date: 当前日期`
- 添加 `deprecated_reason: 废弃原因`
- 移至 INDEX.md 的 Deprecated 区域

**deprecated 的作用**：
1. **明确标记**：表示未完成的工作被主动废弃
2. **可追溯**：保留废弃原因，方便查阅
3. **缓冲期**：30 天后才归档，给用户反悔机会

**示例场景**：
```
用户: 这个需求取消了，废弃相关文档
→ AI: 将文档标记为 deprecated，记录原因
→ 30 天后归档

用户: 这个功能已上线完成了，废弃它
→ AI: completed 状态不能废弃，只能归档
→ 建议：等待30天自动归档，或直接归档
```

---

#### any → archived（自动归档）

**触发时机**：
- **时间到期**：任何状态的文档超过 30 天
- **用户明确归档**：用户说"可以归档"、"移至历史"

**执行动作**：
- 更新 `status: archived`
- 添加 `archived_date: 当前日期`
- 移动文件：`docs/{类型}/archive/{原文件名}-v{版本号}.md`
- 更新 INDEX.md，移至 Archived 区域

**归档后的处理**：
- 保留 6 个月供查阅
- 6 个月后可删除（需用户确认）

**归档周期统一**：
- draft: 创建后 30 天 → archived
- active: 最后更新后 30 天 → archived
- completed: 完成后 30 天 → archived
- deprecated: 废弃后 30 天 → archived

---

### 4. 状态转换流程图

```
┌─────────┐  用户确认   ┌────────┐  功能完成  ┌───────────┐
│  draft  │ ─────────→ │ active │ ─────────→ │ completed │
└─────────┘            └────────┘            └───────────┘
     │                      │                      │
     │ 明确废弃              │ 明确废弃              │
     ↓                      ↓                      │
┌────────────┐        ┌────────────┐              │
│ deprecated │        │ deprecated │              │
└────────────┘        └────────────┘              │
     │                      │                      │
     └──────────────────────┴──────────────────────┘
                              │
                              │ 所有状态 30天后自动归档
                              ↓
                        ┌──────────┐
                        │ archived │ ← 历史记录（6个月）
                        └──────────┘
                              │
                              │ 6个月后
                              ↓
                           [删除]
```

**关键规则**：
- 新版本发布**不改变**旧版本状态
- deprecated 仅用于 **draft/active 状态**
- **completed 不进入 deprecated**，只能直接归档
- 所有状态**统一 30 天归档**

### 5. 状态判断的 AI 指令

**判断当前文档状态**：

```markdown
## 文档状态检查

1. 检查元数据中的 status 字段
2. 根据状态执行对应逻辑：
   - draft: 提示用户确认，检查完整性
   - active: 正常使用，30天后归档
   - completed: 提示已完成，不再变更，30天后归档
   - deprecated: 提示已废弃，记录原因，30天后归档
   - archived: 提示仅作历史参考
```

**状态转换触发条件**：

| 转换 | 触发条件 | AI 动作 |
|-----|---------|---------|
| draft → active | 用户确认 + 完整性检查通过 | 更新状态，记录索引 |
| active → completed | 功能上线/修复合并/实施完成 | 更新状态，记录完成证据 |
| draft → deprecated | 用户明确要求废弃 | 更新状态，记录原因 |
| active → deprecated | 用户明确要求废弃 | 更新状态，记录原因 |
| any → archived | 超过30天 / 用户确认归档 | 移动文件，更新索引 |

**新版本处理规则**：

```
创建新版本时：
- 创建新文档，状态为 draft
- 旧版本文档保持原状态不变
- 不自动废弃旧版本
```

**completed 状态特殊规则**：

```
completed 状态不能进入 deprecated：
- 用户要求废弃 completed 文档时，提示"已完成文档不能废弃"
- 建议：等待30天自动归档，或用户确认直接归档
```

## 二、文档质量检查机制

### 1. 写入时检查（AI 执行）

每次创建文档时必须检查：

| 检查项 | 要求 | 不合格处理 |
|--------|------|-----------|
| 路径规范 | 符合 `{目录}/{日期}-{主题}-{类型}.md` | 拒绝创建，提示正确路径 |
| 元数据完整 | 包含 status/created/updated | 自动补充 |
| 无占位符 | 无 TBD/TODO/待补充/XXX | 拒绝创建，要求完成 |
| 关联文档 | 需求→设计→详细设计关联 | 提示补充关联 |

### 2. 定期审计（每 10 轮对话）

触发 `attention-maintenance` 时执行：

```markdown
## 文档审计清单

- [ ] 所有 active 文档是否与代码实现一致
- [ ] 是否有超过 30 天的文档（应归档）
- [ ] 是否有 draft 状态文档超过 7 天未完成
- [ ] 是否有 deprecated 文档超过 30 天未归档
```

### 3. 代码变更同步

**规则**：任何代码变更必须同步更新文档

| 变更类型 | 文档同步要求 |
|---------|-------------|
| 接口变更 | 更新详细设计文档 |
| 数据库变更 | 更新数据库设计章节 |
| 业务逻辑变更 | 更新设计说明书 |
| Bug 修复 | 更新问题分析文档（添加修复记录） |

**执行流程**：
```
代码变更 → 检查关联文档 → 提示更新文档 → 用户确认 → 同步更新
```

## 三、文档质量控制标准

### 1. 内容完整性

| 文档类型 | 必需章节 |
|---------|---------|
| 需求说明书 | 背景、目标、范围、功能清单、验收标准 |
| 设计说明书 | 架构图、模块划分、技术选型、接口定义 |
| 详细设计 | 类图、时序图、数据结构、核心逻辑 |
| 变更计划 | 变更原因、影响范围、改动要点、回滚策略 |
| 问题分析 | 现象、根因、修复方案、防御措施 |

### 2. 一致性检查

| 检查维度 | 检查方法 |
|---------|---------|
| 需求-设计一致 | 设计说明书覆盖所有需求点 |
| 设计-实现一致 | 代码实现符合设计文档描述 |
| 接口一致 | 文档中的接口定义与代码一致 |
| 命名一致 | 文档中的术语与代码命名一致 |

### 3. 可读性标准

- ✅ 使用清晰的标题层级（不超过 4 级）
- ✅ 每个章节有明确的主题句
- ✅ 复杂概念有图表辅助
- ✅ 代码示例有注释说明
- ❌ 避免大段未分段的文本
- ❌ 避免中英文混杂的术语

## 四、长期维护策略

### 1. 版本控制

**文档版本号规则**：`v{大版本}.{小版本}.{修订号}`

| 版本类型 | 变更程度 | 示例 |
|---------|---------|------|
| 大版本 | 架构调整、重大变更 | v2.0.0 |
| 小版本 | 功能新增、章节扩充 | v1.1.0 |
| 修订号 | 错误修正、措辞调整 | v1.0.1 |

**版本记录**：
```markdown
## 变更记录

| 版本 | 日期 | 变更内容 | 作者 |
|-----|------|---------|------|
| v1.1.0 | 2026-06-23 | 新增权限控制章节 | AI |
| v1.0.1 | 2026-06-20 | 修正接口参数描述 | AI |
| v1.0.0 | 2026-06-15 | 初始版本 | AI |
```

### 2. 过期预警

**检查规则**：
- draft 文档超过 7 天未完成 → 提示完成或删除
- 任何文档超过 30 天 → 自动归档提示
- archived 文档超过 6 个月 → 可删除提示

### 3. 冗余清理

**清理规则**：
- archived 文档超过 6 个月 → 可删除
- 被替代的旧版本 → 保留最新 2 个版本
- 无关联的孤立文档 → 标记待清理

## 五、AI 辅助质量保障

### 1. 创建文档时

```
自动执行：
1. 检查路径规范
2. 生成元数据（status: draft）
3. 验证必需章节
4. 建立文档关联
5. 记录到文档索引
```

### 2. 更新文档时

```
自动执行：
1. 更新 updated 时间
2. 检查一致性影响
3. 同步关联文档
4. 记录变更内容
```

### 3. 定期审计时

```
自动执行：
1. 检查过期文档（超过30天）
2. 发现孤立文档
3. 验证一致性
4. 生成审计报告
```
