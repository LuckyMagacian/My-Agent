# 需求 vs 变更 - 场景区分指南

## 快速判断表

| 判断维度 | 需求开发 | 功能变更 |
|---------|---------|---------|
| **代码状态** | 从零开始 | 已有代码基础 |
| **模块性质** | 新增独立模块 | 修改现有模块 |
| **文档路径** | `docs/requirements/` | `docs/changes/` |
| **前置依赖** | 无历史约束 | 需分析现有实现 |

---

## 关键区分规则

### ✅ 走「需求开发」的场景

1. **新增独立模块/服务/页面**
   - 示例：新增「用户中心」模块
   - 特征：全新目录、独立入口、完整功能闭环

2. **全新功能栈**
   - 示例：引入消息推送系统
   - 特征：需要设计架构、选型、接口定义

3. **无历史代码约束**
   - 不需要考虑现有实现的兼容性
   - 可以自由设计数据结构和接口

**判断口诀**：**新模块、新架构、无历史 → 需求**

---

### ✅ 走「功能变更」的场景

1. **修改已有模块的行为**
   - 示例：调整登录流程、修改表单验证规则
   - 特征：在现有代码上改动

2. **小范围改动**
   - 示例：按钮文案调整、接口字段增减
   - 特征：改动文件 < 5 个，不涉及架构

3. **需要向后兼容**
   - 示例：保留旧接口、数据迁移
   - 特征：需要考虑现有用户/数据

**判断口诀**：**改现有、小范围、需兼容 → 变更**

---

## 边界情况处理

### 场景 1：在大模块中新增子功能

```
用户管理模块（已有）
├── 用户列表 ✅ 已有
├── 用户详情 ✅ 已有
└── 用户权限设置 ❓ 新增
```

**判断**：
- 如果子功能独立性强（新页面、新 API）→ **需求**（子模块级别）
- 如果子功能依附于现有功能（新增字段、新增按钮）→ **变更**

**文档关联示例**：
```
需求：2026-06-23-用户权限-需求说明书.md
设计：2026-06-23-用户权限-设计说明书.md
详细：2026-06-23-用户权限-auth-详细设计.md
详细：2026-06-23-用户权限-api-详细设计.md
```

### 场景 2：重构现有模块

```
将登录模块从 Session 改为 JWT
```

**判断**：
- 重构涉及架构调整、接口变更 → **需求**（但需标注「重构」）
- 如果只是内部实现调整，对外接口不变 → **变更**

### 场景 3：修复 Bug 时发现需要调整设计

```
修复登录超时问题 → 发现需要调整 token 刷新机制
```

**判断**：
- 先走 **问题修复** 流程
- 如果修复需要设计调整 → 补充 **变更计划** 文档

## 模型判断提示词

在 SKILL.md 中增加判断指令：

```markdown
## 模式判断（增强版）

### 第一步：识别关键信号

用户输入中包含以下关键词 → 建议走对应流程：

| 关键词模式 | 推荐流程 |
|-----------|---------|
| 「新增」「开发」「实现」「从零」 | 需求开发 |
| 「修改」「调整」「优化」「重构」 | 功能变更 |
| 「修复」「解决」「Bug」「问题」 | 问题修复 |
| 「分析」「理解」「梳理」「调研」 | 技术分析 |
| 「方案」「细化」「落地」「实施计划」 | 方案优化 |

### 第二步：验证判断

问自己 3 个问题：

1. **是否有现有代码？**
   - 无 → 需求
   - 有 → 继续

2. **改动范围是否 < 20% 模块代码？**
   - 是 → 变更
   - 否 → 继续

3. **是否涉及架构/数据结构变化？**
   - 是 → 需求（重构）
   - 否 → 变更

### 第三步：确认路径

- 需求 → `docs/requirements/`
- 变更 → `docs/changes/`
- 问题 → `docs/bugs/`
- 分析 → `docs/analysis/`
- 方案 → `docs/solution-optimization/`
```

## 实际案例对比

### 案例 1：新增「评论功能」

**判断：需求**
- 原因：新增独立模块
- 文档：`docs/requirements/2026-06-23-评论功能-需求说明书.md`

### 案例 2：修改「评论功能 - 增加表情」

**判断：变更**
- 原因：在现有模块上扩展
- 文档：`docs/changes/2026-06-23-评论增加表情.md`

### 案例 3：修复「评论无法删除」

**判断：问题修复**
- 原因：修复异常行为
- 文档：`docs/bugs/2026-06-23-评论无法删除.md`

## 文档内容差异

| 维度 | 需求说明书 | 变更计划 |
|-----|-----------|---------|
| 背景 | 业务背景、用户价值 | 现有实现、改动原因 |
| 范围 | 功能边界、验收标准 | 影响范围、改动文件 |
| 设计 | 架构设计、接口定义 | 改动要点、兼容策略 |
| 风险 | 技术风险、依赖风险 | 回归风险、数据迁移风险 |
| 回滚 | 无（新功能） | 回滚策略、版本要求 |
