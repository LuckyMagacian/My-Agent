# 软件开发与变更流程

## 核心原则：文档先行，依照文档执行

**工作流程**：先产出文档 → 用户确认文档 → 严格按文档执行

**强制规则**：
1. **文档是执行依据**：任何编码、修改、修复动作，必须有对应文档作为依据
2. **先写后做**：禁止直接写代码，必须先完成需求/设计/计划文档
3. **按文档执行**：执行时必须对照文档内容，不得随意偏离
4. **偏离需确认**：发现文档有误或需调整，必须暂停执行，更新文档并获得用户确认后再继续

**文档路径**：`{项目根目录}/docs/`

| 场景 | 文档路径 | 作用 |
|------|----------|------|
| 需求 | `docs/需求说明书.md` | 定义做什么 |
| 设计 | `docs/设计说明书.md` | 定义怎么做 |
| 详细设计 | `docs/design/{模块}.md` | 定义具体实现 |
| 变更计划 | `docs/changes/{日期}-{主题}.md` | 定义改什么 |
| 问题修复 | `docs/bugs/{日期}-{问题}.md` | 定义修什么 |
| 分析报告 | `docs/analysis/{模块}-分析报告.md` | 记录分析结果 |
| 方案优化 | `docs/solution-optimization/{日期}-{主题}.md` | 定义优化路径 |

---

## 模式判断

| 工作流 | 适用场景 |
| --- | --- |
| **完整需求开发** | 新增独立模块/页面/服务，需要需求→设计→实现全流程 |
| **功能变更** | 已有项目中的小范围改动、局部功能修改、样式调整 |
| **问题修复** | 修复已有功能的异常行为、Bug、逻辑错误、状态不一致 |
| **技术与代码分析** | 理解现有代码的实现原理、架构设计、技术方案，不做修改 |
| **方案优化** | 已有技术方案需要结合需求和项目信息，给出更明确、更具体、可分步实现的优化方案 |

**判断原则**：改动涉及新增独立模块走完整需求；修复已有功能的异常走问题修复；用户意图是理解而非修改走技术分析；已有方案需细化落地走方案优化；其余走功能变更。


> 详细步骤参见同目录 `workflow-details.md`。

## 一、完整需求开发

```
需求文档 → 用户确认 → 设计文档 → 用户确认 → 详细设计 → 用户确认 → 按文档编码
```

| 阶段 | 核心动作 | 输出文档 | 执行依据 |
| --- | --- | --- | --- |
| 1. 需求开发 | 提问+选项收集、分类结构化、MoSCoW优先级 | `docs/需求说明书.md` | 定义做什么 |
| 2. 概要设计 | 领域分析、架构设计、技术选型 | `docs/设计说明书.md` | 定义怎么做 |
| 3. 详细设计 | 接口/数据库/核心逻辑/前端设计 | `docs/design/{模块}.md` | 定义具体实现 |
| 4. 实施编码 | 后端→前端、Domain→Infra→App→Interface | 按前三份文档执行 | 对照文档编码 |

**编码顺序**：Domain层 → Infrastructure层 → Application层 → Interface层 → 前端

**执行规则**：
- 编码前必须阅读需求文档、设计文档、详细设计文档
- 每写一个模块，对照详细设计文档确认接口、数据结构、逻辑流程
- 发现文档有误或需调整，立即暂停，更新文档并获得用户确认后再继续

**⚠️ 强制规则：文档就绪后等待用户确认，收到确认后方可进入下一阶段。**

## 二、功能变更

```
变更计划文档 → 用户确认 → 按文档执行变更
```

**原则**：最小改动 | 向后兼容 | 架构一致 | 可回退 | 渐进式

**执行流程**：
1. 需求澄清后，产出变更计划文档 `docs/changes/{日期}-{变更主题}.md`
2. 文档内容：变更范围、影响文件、改动要点、回退策略
3. 用户确认文档后，严格按文档执行变更
4. 执行时对照文档，每完成一项改动勾选确认

**⚠️ 未产出文档且未收到确认，禁止开始实施变更。**

---

## 三、问题修复

```
问题分析文档 → 用户确认 → 按文档执行修复
```

**原则**：先理解后修复 | 根因修复 | 最小侵入 | 防御性加固 | 回归验证

**执行流程**：
1. 问题分析后，产出问题分析文档 `docs/bugs/{日期}-{问题摘要}.md`
2. 文档内容：问题现象、根因分析、影响范围、修复方案、防御措施
3. 用户确认文档后，严格按文档执行修复
4. 执行时对照文档，每完成一项修复勾选确认

**⚠️ 未产出文档且未收到确认，禁止开始实施修复。**

---

## 四、技术与代码分析

```
分析过程 → 输出分析报告文档
```

**原则**：先整体后局部 | 先意图后实现 | 只读分析 | 证据链支撑 | 分层递进

**执行流程**：
1. 分析完成后，产出分析报告文档 `docs/analysis/{模块}-分析报告.md`
2. 文档内容：架构概览、核心流程、技术要点、设计亮点、潜在问题、改进建议
3. 用户确认文档后，流程结束

**⚠️ 本流程为只读分析，不修改任何代码。**

---

## 五、方案优化

```
需求理解 → 现状分析 → 方案对比 → 优化设计 → 用户确认 → 分步实施计划
```

**原则**：需求驱动 | 现状约束 | 方案可比 | 步骤可执行 | 渐进落地

**适用场景**：
- 已有技术方案过于笼统，需要细化到可执行步骤
- 多个候选方案需要对比决策
- 方案需要结合项目实际约束（技术栈、团队能力、时间）调整
- 将高层设计转化为可分步实施的落地计划

**执行流程**：

| 阶段 | 核心动作 | 输出 |
| --- | --- | --- |
| 1. 需求理解 | 梳理原始需求、业务目标、验收标准 | 明确优化目标 |
| 2. 现状分析 | 分析现有方案、技术约束、项目上下文 | 识别约束边界 |
| 3. 方案对比 | 提出多个候选方案，从可行性、成本、风险维度对比 | 方案对比矩阵 |
| 4. 优化设计 | 融合最优方案，输出详细实施步骤 | `docs/solution-optimization/{日期}-{主题}.md` |
| 5. 分步实施 | 按确认后的方案，拆分为可独立交付的阶段 | 阶段性 Checklist |

**⚠️ 方案文档必须经用户确认后，方可进入实施阶段。实施时严格按分步计划执行。**

> 详细步骤参见同目录 `workflow-details.md`。

## 执行对照机制

**执行时必须对照文档**：
- 每开始一个任务，先阅读对应文档
- 每完成一个步骤，对照文档确认是否符合
- 发现偏离文档，立即暂停并报告

**偏离处理**：
1. 暂停当前执行
2. 说明偏离原因
3. 提出文档调整建议
4. 获得用户确认
5. 更新文档
6. 按更新后的文档继续执行

---

## 通用约束

1. **文档先行**：任何编码动作必须有对应文档作为依据
2. **按文档执行**：执行时必须对照文档，不得随意偏离
3. **增量演进**：每步完成后验证，避免大批量改动后才发现错误
4. **变更可追溯**：所有文档变更须标注变更内容和原因
5. **错误处理**：任何阶段的实现都必须包含错误处理设计

## 文档检查清单（每个阶段结束时必须执行）

- [ ] 文档已创建在 `docs/` 对应子目录下
- [ ] 文档内容完整，无 TBD/TODO/待补充
- [ ] 用户已确认文档内容
- [ ] 确认后方可进入下一阶段
