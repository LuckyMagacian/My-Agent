# 软件开发与变更流程

## 文档化强制规则

**文档路径**：`{项目根目录}/docs/`

**硬性检查点**：每个阶段结束时，必须输出对应文档到 `docs/` 目录，否则**禁止进入下一阶段**。

**文档检查清单**：
- [ ] 文档已创建在 `docs/` 目录下
- [ ] 文档内容完整（无 TBD/TODO/待补充）
- [ ] 用户已确认文档内容

---

## 模式判断

| 工作流 | 适用场景 |
| --- | --- |
| **完整需求开发** | 新增独立模块/页面/服务，需要需求→设计→实现全流程 |
| **功能变更** | 已有项目中的小范围改动、局部功能修改、样式调整 |
| **问题修复** | 修复已有功能的异常行为、Bug、逻辑错误、状态不一致 |
| **技术与代码分析** | 理解现有代码的实现原理、架构设计、技术方案，不做修改 |

**判断原则**：改动涉及新增独立模块走完整需求；修复已有功能的异常走问题修复；用户意图是理解而非修改走技术分析；其余走功能变更。


> 详细步骤参见同目录 `workflow-details.md`。

---

## 一、完整需求开发

```
需求收集 → 用户确认 → 概要设计 → 用户确认 → 详细设计 → 用户确认 → 实施编码
```

| 阶段 | 核心动作 | 输出位置 | 强制检查点 |
| --- | --- | --- | --- |
| 1. 需求开发 | 提问+选项收集、分类结构化、MoSCoW优先级 | `docs/需求说明书.md` | ✅ 文档存在 + 用户确认 |
| 2. 概要设计 | 领域分析、架构设计、技术选型 | `docs/设计说明书.md` | ✅ 文档存在 + 用户确认 |
| 3. 详细设计 | 接口/数据库/核心逻辑/前端设计 | `docs/design/{模块}.md` | ✅ 文档存在 + 用户确认 |
| 4. 实施编码 | 后端→前端、Domain→Infra→App→Interface | 代码文件 | ✅ 编译通过 |

**编码顺序**：Domain层 → Infrastructure层 → Application层 → Interface层 → 前端

**⚠️ 强制规则：每个阶段结束后必须输出文档到 `docs/` 目录，文档就绪后等待用户确认，收到确认后方可进入下一阶段。**

---

## 二、功能变更

```
需求澄清 → 制定计划 → 用户确认 → 实施变更
```

**原则**：最小改动 | 向后兼容 | 架构一致 | 可回退 | 渐进式

**强制文档输出**：
- 制定计划阶段：必须输出 `docs/changes/{日期}-{变更主题}.md`
- 文档内容：变更范围、影响文件、改动要点、回退策略

**⚠️ 计划制定后必须等待用户确认再执行。未输出文档不得进入实施阶段。**

---

## 三、问题修复

```
问题分析 → 影响范围分析 → 制定修复方案 → 用户确认 → 实施修复
```

**原则**：先理解后修复 | 根因修复 | 最小侵入 | 防御性加固 | 回归验证

**强制文档输出**：
- 问题分析阶段：必须输出 `docs/bugs/{日期}-{问题摘要}.md`
- 文档内容：问题现象、根因分析、影响范围、修复方案、防御措施

**⚠️ 方案制定后必须等待用户确认再执行。未输出文档不得进入实施阶段。**

---

## 四、技术与代码分析

```
范围确认 → 宏观架构分析 → 核心流程追踪 → 关键技术点分析 → 问题与亮点识别 → 分析报告输出
```

**原则**：先整体后局部 | 先意图后实现 | 只读分析 | 证据链支撑 | 分层递进

**强制文档输出**：
- 分析完成后：必须输出 `docs/analysis/{模块}-分析报告.md`
- 文档内容：架构概览、核心流程、技术要点、设计亮点、潜在问题、改进建议

**⚠️ 本流程为只读分析，不修改任何代码。分析报告必须输出到 `docs/analysis/` 目录。**

---

## 通用约束

1. **文档先行**：设计和需求必须先文档化，经用户确认后再编码
2. **增量演进**：每步完成后验证，避免大批量改动后才发现错误
3. **变更可追溯**：所有文档变更须标注变更内容和原因
4. **错误处理**：任何阶段的实现都必须包含错误处理设计
5. **文档路径强制**：所有文档必须输出到 `{项目根目录}/docs/` 目录下，子目录结构：
   - `docs/` - 需求、设计文档
   - `docs/design/` - 详细设计
   - `docs/changes/` - 功能变更计划
   - `docs/bugs/` - 问题修复记录
   - `docs/analysis/` - 技术分析报告

## 文档化检查清单（每个阶段结束时必须执行）

- [ ] 文档已创建在 `docs/` 对应子目录下
- [ ] 文档内容完整，无 TBD/TODO/待补充
- [ ] 用户已确认文档内容
- [ ] 确认后方可进入下一阶段
