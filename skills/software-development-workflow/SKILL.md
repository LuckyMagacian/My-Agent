# 软件开发与变更流程

## 模式判断

| 工作流 | 适用场景 |
| --- | --- |
| **完整需求开发** | 新增独立模块/页面/服务，需要需求→设计→实现全流程 |
| **功能变更** | 已有项目中的小范围改动、局部功能修改、样式调整 |
| **问题修复** | 修复已有功能的异常行为、Bug、逻辑错误、状态不一致 |
| **技术与代码分析** | 理解现有代码的实现原理、架构设计、技术方案，不做修改 |

**判断原则**：改动涉及新增独立模块走完整需求；修复已有功能的异常走问题修复；用户意图是理解而非修改走技术分析；其余走功能变更。


> 详细步骤参见同目录 `workflow-details.md`。

---

## 一、完整需求开发

```
需求收集 → 用户确认 → 概要设计 → 用户确认 → 详细设计 → 用户确认 → 实施编码
```

| 阶段 | 核心动作 | 输出位置 |
| --- | --- | --- |
| 1. 需求开发 | 提问+选项收集、分类结构化、MoSCoW优先级 | `{docs}/需求说明书.md` |
| 2. 概要设计 | 领域分析、架构设计、技术选型 | `{docs}/设计说明书.md` |
| 3. 详细设计 | 接口/数据库/核心逻辑/前端设计 | `{docs}/design/{模块}.md` |
| 4. 实施编码 | 后端→前端、Domain→Infra→App→Interface | 代码文件 |

**编码顺序**：Domain层 → Infrastructure层 → Application层 → Interface层 → 前端

**⚠️ 每个阶段结束后必须等待用户确认再进入下一阶段。**

---

## 二、功能变更

```
需求澄清 → 制定计划 → 用户确认 → 实施变更
```

**原则**：最小改动 | 向后兼容 | 架构一致 | 可回退 | 渐进式

**⚠️ 计划制定后必须等待用户确认再执行。**

---

## 三、问题修复

```
问题分析 → 影响范围分析 → 制定修复方案 → 用户确认 → 实施修复
```

**原则**：先理解后修复 | 根因修复 | 最小侵入 | 防御性加固 | 回归验证

**⚠️ 方案制定后必须等待用户确认再执行。**

---

## 四、技术与代码分析

```
范围确认 → 宏观架构分析 → 核心流程追踪 → 关键技术点分析 → 问题与亮点识别 → 分析报告输出
```

**原则**：先整体后局部 | 先意图后实现 | 只读分析 | 证据链支撑 | 分层递进

**⚠️ 本流程为只读分析，不修改任何代码。**

---

## 通用约束

1. **文档先行**：设计和需求必须先文档化，经用户确认后再编码
2. **增量演进**：每步完成后验证，避免大批量改动后才发现错误
3. **变更可追溯**：所有文档变更须标注变更内容和原因
4. **错误处理**：任何阶段的实现都必须包含错误处理设计
5. **文档路径**：优先使用项目已有的文档目录结构；若无则创建 `docs/`
