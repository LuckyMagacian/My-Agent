# 工作流程详细步骤参考

## 一、完整需求开发流程

### 阶段 1：需求开发
**目标**：与用户充分沟通，形成结构化、可追溯的需求文档。

1. **需求收集**
    - 通过提问 + 提供选项的方式与用户沟通，避免遗漏关键需求
    - 对模糊需求主动追问，给出可选方案并标注推荐项
    - 分析项目中已有的相关模块和可复用组件
2. **需求分类与结构化**
    - **功能性需求**：核心业务功能、用户交互流程
    - **非功能性需求**：性能、安全、可用性、可维护性、可观测性
    - **接口需求**：IPC接口、事件接口、外部系统集成
    - **数据需求**：数据模型、存储方案、数据生命周期
    - 每条需求须包含：唯一标识（如 `REQ-001`）、优先级（MoSCoW）、验收标准
3. **文档输出** → 项目文档目录下创建需求文档（增量更新并标注变更版本）

**质量检查**：每条需求有唯一标识和优先级 | 非功能性需求已覆盖

---

### 阶段 2：概要方案设计
**目标**：基于DDD理念，设计系统架构方案。

1. **领域分析**：识别限界上下文、定义聚合根/实体/值对象、建立统一语言术语表、绘制上下文映射图
2. **架构设计**：DDD四层分层（Interface → Application → Domain → Infrastructure）、组件划分与职责、交互模式、数据架构
3. **技术选型**：列出语言/框架/中间件清单，说明选型理由和权衡取舍
4. **文档输出** → 项目文档目录下创建设计文档

**质量检查**：聚合边界清晰 | 分层职责明确

---

### 阶段 3：详细方案设计
**目标**：将概要设计细化到模块/类级别，可直接指导编码。

1. **接口设计**：API契约、DTO定义、IPC Command定义
2. **数据库设计**：ER图、表结构、索引策略、迁移脚本
3. **核心逻辑设计**：算法流程、状态机、领域事件、错误处理策略
4. **前端架构设计**：组件树、状态管理、路由导航
5. **文档输出** → 项目文档目录下按模块分文件

**质量检查**：接口入参/出参/异常明确 | 表结构含字段类型和约束 | 关键流程有时序图

---

### 阶段 4：实施编码
**实施顺序（严格遵循）**：

1. **先后端，后前端**
2. **后端内部**：Domain层 → Infrastructure层 → Application层 → Interface层
3. **先静态，后动态**（先定义结构/类型，再实现逻辑）
4. **先结构，后内容，再样式**（前端UI开发顺序）

**编码规范**：
- Application层只做编排，不写业务规则
- Domain层不依赖Infrastructure层（依赖倒置）
- Infrastructure层实现Repository接口
- 每完成一个模块须验证编译通过后再继续

---

## 二、功能变更工作流程

**遵循原则**：
1. **最小改动原则**：只修改与变更需求直接相关的代码，不做无关重构
2. **向后兼容原则**：变更不得破坏已有功能的正常行为
3. **架构一致性原则**：遵循现有架构风格、分层规范和命名约定
4. **可回退原则**：变更方案须考虑回退策略
5. **渐进式变更原则**：优先拆分为多个小步骤逐步交付

### 步骤 1：需求澄清
- 通过提问 + 提供选项与用户沟通
- 明确变更范围、影响面、预期行为
- 分析相关代码和可复用组件，识别潜在风险

### 步骤 2：制定计划
- 生成具体改动计划，列出需修改的文件和改动要点
- 保存到项目文档目录

### 步骤 3：用户确认
- **必须**要求用户查看改动计划
- 用户确认后方可执行
- **未收到确认指令前，不得开始实施变更**

### 步骤 4：实施变更
- 严格按确认后的计划执行
- 每完成一项改动验证编译/运行是否正常
- 发现计划外问题及时与用户沟通调整

---

## 三、问题修复(bugfix)工作流程

**遵循原则**：
1. **先理解后修复原则**：修复前必须充分理解现有代码的设计意图和上下文
2. **根因修复原则**：必须定位并修复根本原因，禁止仅做表面症状的 workaround
3. **最小侵入原则**：优先选择影响范围最小、改动量最少的修复方案
4. **防御性加固原则**：修复时应评估并增加防御性代码，防止同类问题再次发生
5. **回归验证原则**：修复后必须对影响范围内的相关功能进行验证
 
### 步骤 1：问题分析 请严格遵循以下步骤执行：
- **时序重构**：将提供的日志按时间线排列，识别 IPC、API 调用与报错发生的先后逻辑关系。
- **副作用评估**：分析报错代码段（如 Slate 的 findPath）在 React 生命周期中的执行时机，寻找可能导致数据模型与视图不同步的源头（如：initialValue 的注入时机）。
- **状态断言**：推导出在该错误发生前，编辑器内部的 JSON 状态树必须满足什么约束条件？当前日志显示的状态是否违背了这些约束？
- **复现问题**：明确问题的触发条件、现象和复现步骤
- **定位根因**：通过日志、调试、代码追踪等手段，定位根本原因
- **归类问题**：逻辑错误 / 状态不一致 / 数据异常 / 接口契约不匹配 / 边界条件遗漏 / 并发竞态
- **记录发现**：将根因分析结论写入项目文档目录

### 步骤 2：影响范围分析
- **直接受影响**：列出修复直接修改的文件、函数、数据结构
- **间接受影响**：分析调用链上下游、依赖该模块的其他功能
- **回归风险评估**：评估修复是否可能引入新问题

### 步骤 3：制定修复方案
- **方案设计**：基于根因和影响范围提出修复方案
- **备选方案**：多种修复路径时列出优劣对比，标注推荐项
- **防御性加固**：评估是否需要增加防御性代码

### 步骤 4：用户确认
- **必须**要求用户查看修复方案及影响分析
- 用户确认后方可执行
- **未收到确认指令前，不得开始实施修复**

### 步骤 5：实施修复
- 严格按确认后的方案执行
- 每完成一项改动验证编译/运行是否正常
- 修复完成后进行回归验证，确保未引入新问题

## 四、技术与代码分析工作流程

**遵循原则**：
1. **先整体后局部原则**：先理解系统/模块全貌，再深入具体实现细节
2. **先意图后实现原则**：先理解"为什么这样做"，再分析"怎么做的"
3. **最小侵入原则**：只读分析，不修改任何代码或文档
4. **证据链原则**：每个结论必须有代码位置或文档引用作为支撑
5. **分层递进原则**：按架构分层逐层分析，避免跳跃式理解

### 步骤 1：范围确认
- 通过提问 + 提供选项与用户沟通
- 明确分析对象：整个系统 / 特定模块 / 特定流程 / 特定技术点
- 明确分析深度：概览级 / 模块级 / 代码级
- 明确分析目的：技术选型评估 / 学习参考 / 二次开发准备 / 技术评审

### 步骤 2：宏观架构分析
- **技术栈识别**：语言、框架、依赖库、构建工具
- **分层结构**：识别项目的架构模式（MVC/DDD/微服务等）
- **模块划分**：梳理核心模块及其职责边界
- **依赖关系**：模块间的调用关系和数据流向
- **输出**：架构概览图（文字描述）

### 步骤 3：核心流程追踪
- **入口识别**：定位关键流程的触发点（API入口/事件监听/定时任务）
- **调用链追踪**：沿调用链逐层深入，记录关键节点
- **数据流分析**：数据如何在各层/模块间流转和变换
- **状态变化**：关键状态机和状态转换逻辑
- **输出**：核心流程时序描述

### 步骤 4：关键技术点分析
- **设计模式**：识别使用的设计模式及其适用场景
- **核心算法**：分析关键算法的实现和复杂度
- **并发/异步**：并发控制、异步处理机制
- **错误处理**：异常捕获、降级、重试策略
- **扩展机制**：插件、钩子、中间件等扩展点
- **输出**：技术要点清单

### 步骤 5：问题与亮点识别
- **设计亮点**：值得学习和复用的优秀设计
- **潜在问题**：代码异味、架构债务、性能隐患
- **改进建议**：如有明显可优化点，列出建议（仅建议，不执行）
- **输出**：评价与建议清单

### 步骤 6：分析报告输出
- 汇总分析结果，形成结构化报告
- 保存到项目文档目录（如 `docs/analysis/`）
- 包含：架构概览、核心流程、技术要点、评价建议

**⚠️ 本流程为只读分析，不修改任何代码。如用户后续需要基于分析结果进行变更，应切换到对应的工作流。**

**文档输出规范**：

| 内容 | 输出位置 | 格式要求 |
| --- | --- | --- |
| 架构概览 | `{docs}/analysis/{模块}-架构概览.md` | 模块图 + 职责说明 |
| 流程分析 | `{docs}/analysis/{流程名}-流程分析.md` | 时序描述 + 关键代码引用 |
| 技术要点 | `{docs}/analysis/{模块}-技术要点.md` | 设计模式 + 算法 + 机制说明 |
| 综合报告 | `{docs}/analysis/{模块}-分析报告.md` | 汇总所有分析内容 |
