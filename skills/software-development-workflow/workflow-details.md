# 工作流程详细步骤参考

## 核心原则：文档先行，依照文档执行

**工作流程**：先产出文档 → 用户确认文档 → 严格按文档执行

**强制规则**：
1. **文档是执行依据**：任何编码、修改、修复动作，必须有对应文档作为依据
2. **先写后做**：禁止直接写代码，必须先完成需求/设计/计划文档
3. **按文档执行**：执行时必须对照文档内容，不得随意偏离
4. **偏离需确认**：发现文档有误或需调整，必须暂停执行，更新文档并获得用户确认后再继续

**文档路径**：`{项目根目录}/docs/`

**硬性规则**：每个阶段结束时，必须输出对应文档到 `docs/` 目录，否则**禁止进入下一阶段**。

**文档检查清单**（每个阶段结束时必须执行）：
- [ ] 文档已创建在 `docs/` 对应子目录下
- [ ] 文档内容完整，无 TBD/TODO/待补充
- [ ] 用户已确认文档内容
- [ ] 确认后方可进入下一阶段

## 一、完整需求开发流程

### 阶段 1：需求开发
**目标**：与用户充分沟通，形成结构化、可追溯的需求文档。

1. **需求收集**
    - 通过提问 + 提供选项的方式与用户沟通，避免遗漏关键需求
    - 对模糊需求主动追问，给出可选方案并标注推荐项
    - 分析项目中已有的相关模块和可复用组件
2. **需求分类与结构化**
    - **功能性需求**：核心业务功能、用户交互流程
    - **非功能性需求**：性能、安全、可用性、可维护性、可观测性
    - **接口需求**：IPC接口、事件接口、外部系统集成
    - **数据需求**：数据模型、存储方案、数据生命周期
    - 每条需求须包含：唯一标识（如 `REQ-001`）、优先级（MoSCoW）、验收标准
3. **文档输出** → 必须创建 `docs/需求说明书.md`（增量更新并标注变更版本）

**质量检查**：每条需求有唯一标识和优先级 | 非功能性需求已覆盖

**📋 文档检查点**：
- [ ] `docs/需求说明书.md` 已创建
- [ ] 内容完整，无 TBD/TODO
- [ ] 用户已确认

**⚠️ 未完成文档检查点，禁止进入下一阶段**

---

### 阶段 2：概要方案设计
**目标**：基于DDD理念，设计系统架构方案。

1. **领域分析**：识别限界上下文、定义聚合根/实体/值对象、建立统一语言术语表、绘制上下文映射图
2. **架构设计**：DDD四层分层（Interface → Application → Domain → Infrastructure）、组件划分与职责、交互模式、数据架构
3. **技术选型**：列出语言/框架/中间件清单，说明选型理由和权衡取舍
4. **文档输出** → 必须创建 `docs/设计说明书.md`

**质量检查**：聚合边界清晰 | 分层职责明确

**📋 文档检查点**：
- [ ] `docs/设计说明书.md` 已创建
- [ ] 内容完整，无 TBD/TODO
- [ ] 用户已确认

**⚠️ 未完成文档检查点，禁止进入下一阶段**

---

### 阶段 3：详细方案设计
**目标**：将概要设计细化到模块/类级别，可直接指导编码。

1. **接口设计**：API契约、DTO定义、IPC Command定义
2. **数据库设计**：ER图、表结构、索引策略、迁移脚本
3. **核心逻辑设计**：算法流程、状态机、领域事件、错误处理策略
4. **前端架构设计**：组件树、状态管理、路由导航
5. **文档输出** → 必须创建 `docs/design/{模块}.md` 按模块分文件

**质量检查**：接口入参/出参/异常明确 | 表结构含字段类型和约束 | 关键流程有时序图

**📋 文档检查点**：
- [ ] `docs/design/{模块}.md` 已创建
- [ ] 内容完整，无 TBD/TODO
- [ ] 用户已确认

**⚠️ 未完成文档检查点，禁止进入下一阶段**

---

### 阶段 4：按文档实施编码

**执行前必须阅读**：
- `docs/需求说明书.md` - 确认要做什么
- `docs/设计说明书.md` - 确认架构和选型
- `docs/design/{模块}.md` - 确认具体实现细节

**实施顺序（严格遵循）**：

1. **先后端，后前端**
2. **后端内部**：Domain层 → Infrastructure层 → Application层 → Interface层
3. **先静态，后动态**（先定义结构/类型，再实现逻辑）
4. **先结构，后内容，再样式**（前端UI开发顺序）

**编码规范**：
- Application层只做编排，不写业务规则
- Domain层不依赖Infrastructure层（依赖倒置）
- Infrastructure层实现Repository接口
- 每完成一个模块须验证编译通过后再继续

**按文档执行规则**：
- 每写一个模块，对照详细设计文档确认接口、数据结构、逻辑流程
- 发现文档有误或需调整，立即暂停，更新文档并获得用户确认后再继续
- 禁止偏离文档内容进行编码

## 二、功能变更工作流程

**遵循原则**：
1. **最小改动原则**：只修改与变更需求直接相关的代码，不做无关重构
2. **向后兼容原则**：变更不得破坏已有功能的正常行为
3. **架构一致性原则**：遵循现有架构风格、分层规范和命名约定
4. **可回退原则**：变更方案须考虑回退策略
5. **渐进式变更原则**：优先拆分为多个小步骤逐步交付

### 步骤 1：需求澄清
- 通过提问 + 提供选项与用户沟通
- 明确变更范围、影响面、预期行为
- 分析相关代码和可复用组件，识别潜在风险

### 步骤 2：制定计划
- 生成具体改动计划，列出需修改的文件和改动要点
- **强制输出**：必须创建 `docs/changes/{日期}-{变更主题}.md`

### 步骤 3：用户确认
- **必须**要求用户查看改动计划文档
- 用户确认后方可执行
- **⚠️ 未输出文档且未收到确认指令前，不得开始实施变更**

**📋 文档检查点**：
- [ ] `docs/changes/{日期}-{变更主题}.md` 已创建
- [ ] 内容完整，无 TBD/TODO
- [ ] 用户已确认

### 步骤 4：按文档实施变更
- **执行前必须阅读** `docs/changes/{日期}-{变更主题}.md`
- 严格按文档内容执行，每完成一项改动在文档中勾选确认
- 每完成一项改动验证编译/运行是否正常
- 发现文档有误或需调整，立即暂停，更新文档并获得用户确认后再继续
- 禁止偏离文档内容进行变更

## 三、问题修复(bugfix)工作流程

**遵循原则**：
1. **先理解后修复原则**：修复前必须充分理解现有代码的设计意图和上下文
2. **根因修复原则**：必须定位并修复根本原因，禁止仅做表面症状的 workaround
3. **最小侵入原则**：优先选择影响范围最小、改动量最少的修复方案
4. **防御性加固原则**：修复时应评估并增加防御性代码，防止同类问题再次发生
5. **回归验证原则**：修复后必须对影响范围内的相关功能进行验证
 
### 步骤 1：问题分析 请严格遵循以下步骤执行：
- **时序重构**：将提供的日志按时间线排列，识别 IPC、API 调用与报错发生的先后逻辑关系。
- **副作用评估**：分析报错代码段（如 Slate 的 findPath）在 React 生命周期中的执行时机，寻找可能导致数据模型与视图不同步的源头（如：initialValue 的注入时机）。
- **状态断言**：推导出在该错误发生前，编辑器内部的 JSON 状态树必须满足什么约束条件？当前日志显示的状态是否违背了这些约束？
- **复现问题**：明确问题的触发条件、现象和复现步骤
- **定位根因**：通过日志、调试、代码追踪等手段，定位根本原因
- **归类问题**：逻辑错误 / 状态不一致 / 数据异常 / 接口契约不匹配 / 边界条件遗漏 / 并发竞态
- **记录发现**：必须创建 `docs/bugs/{日期}-{问题摘要}.md`，记录根因分析结论

### 步骤 1.5：信息不足时的升级策略

**触发条件**（满足任一即触发）：
- 当前收集到的信息不足以分析问题根本原因
- 多次尝试修复未能解决问题
- 日志/错误信息不完整，无法还原问题现场

**升级措施**（按优先级执行）：

1. **增加调试代码获取更多信息**
   - 在关键路径添加日志输出（入参、出参、中间状态）
   - 在可疑位置添加断言或条件断点代码
   - 记录添加的调试代码位置和预期观察点
   - **前端专属调试手段**：
     - 添加临时边框/背景色高亮可疑元素（如 `outline: 2px solid red`）
     - 添加调试弹窗显示关键状态/数据（如 `alert` 或自定义 DebugPanel）
     - 在 DOM 元素上添加 `data-debug-*` 属性标记状态
     - 使用 `console.table` 输出结构化数据便于观察
   - 调试代码须为临时性质，修复后必须移除

2. **请求用户提供外部信息**
   - 明确告知用户需要哪些信息，以及为什么需要
   - 可请求的信息类型：
     - 完整的错误日志/控制台输出
     - 无法从项目中获取的代码片段（如第三方服务配置、环境变量）
     - 问题复现的详细步骤和环境信息
     - 相关的截图或录屏
   - 将用户提供的信息记录到 `docs/bugs/{日期}-{问题摘要}.md`

**输出要求**：
- 更新问题分析文档，记录当前分析卡点和已采取的调试措施
- 明确列出还需要哪些信息才能继续分析

---

### 步骤 2：影响范围分析
- **直接受影响**：列出修复直接修改的文件、函数、数据结构
- **间接受影响**：分析调用链上下游、依赖该模块的其他功能
- **回归风险评估**：评估修复是否可能引入新问题

### 步骤 3：制定修复方案
- **方案设计**：基于根因和影响范围提出修复方案
- **备选方案**：多种修复路径时列出优劣对比，标注推荐项
- **防御性加固**：评估是否需要增加防御性代码
- **强制输出**：更新 `docs/bugs/{日期}-{问题摘要}.md`，补充影响范围和修复方案

### 步骤 4：用户确认
- **必须**要求用户查看修复方案及影响分析
- 用户确认后方可执行
- **⚠️ 未输出文档且未收到确认指令前，不得开始实施修复**

**📋 文档检查点**：
- [ ] `docs/bugs/{日期}-{问题摘要}.md` 已创建/更新
- [ ] 包含：问题现象、根因分析、影响范围、修复方案
- [ ] 用户已确认

### 步骤 5：按文档实施修复
- **执行前必须阅读** `docs/bugs/{日期}-{问题摘要}.md`
- 严格按文档中的修复方案执行，每完成一项修复在文档中勾选确认
- 每完成一项改动验证编译/运行是否正常
- 发现文档有误或需调整，立即暂停，更新文档并获得用户确认后再继续
- 修复完成后进行回归验证，确保未引入新问题
- 禁止偏离文档内容进行修复

## 四、技术与代码分析工作流程

**遵循原则**：
1. **先整体后局部原则**：先理解系统/模块全貌，再深入具体实现细节
2. **先意图后实现原则**：先理解"为什么这样做"，再分析"怎么做的"
3. **最小侵入原则**：只读分析，不修改任何代码或文档
4. **证据链原则**：每个结论必须有代码位置或文档引用作为支撑
5. **分层递进原则**：按架构分层逐层分析，避免跳跃式理解

### 步骤 1：范围确认
- 通过提问 + 提供选项与用户沟通
- 明确分析对象：整个系统 / 特定模块 / 特定流程 / 特定技术点
- 明确分析深度：概览级 / 模块级 / 代码级
- 明确分析目的：技术选型评估 / 学习参考 / 二次开发准备 / 技术评审

### 步骤 2：宏观架构分析
- **技术栈识别**：语言、框架、依赖库、构建工具
- **分层结构**：识别项目的架构模式（MVC/DDD/微服务等）
- **模块划分**：梳理核心模块及其职责边界
- **依赖关系**：模块间的调用关系和数据流向
- **输出**：架构概览图（文字描述）

### 步骤 3：核心流程追踪
- **入口识别**：定位关键流程的触发点（API入口/事件监听/定时任务）
- **调用链追踪**：沿调用链逐层深入，记录关键节点
- **数据流分析**：数据如何在各层/模块间流转和变换
- **状态变化**：关键状态机和状态转换逻辑
- **输出**：核心流程时序描述

### 步骤 4：关键技术点分析
- **设计模式**：识别使用的设计模式及其适用场景
- **核心算法**：分析关键算法的实现和复杂度
- **并发/异步**：并发控制、异步处理机制
- **错误处理**：异常捕获、降级、重试策略
- **扩展机制**：插件、钩子、中间件等扩展点
- **输出**：技术要点清单

### 步骤 5：问题与亮点识别
- **设计亮点**：值得学习和复用的优秀设计
- **潜在问题**：代码异味、架构债务、性能隐患
- **改进建议**：如有明显可优化点，列出建议（仅建议，不执行）
- **输出**：评价与建议清单

### 步骤 6：分析报告输出
- 汇总分析结果，形成结构化报告
- **强制输出**：必须创建 `docs/analysis/{模块}-分析报告.md`
- 包含：架构概览、核心流程、技术要点、评价建议

**📋 文档检查点**：
- [ ] `docs/analysis/{模块}-分析报告.md` 已创建
- [ ] 内容完整，包含所有分析结果
- [ ] 用户已确认

**⚠️ 本流程为只读分析，不修改任何代码。如用户后续需要基于分析结果进行变更，应切换到对应的工作流。**

## 五、方案优化工作流程

**遵循原则**：
1. **需求驱动原则**：优化目标必须来源于明确的业务需求或技术痛点，不做无目标的优化
2. **现状约束原则**：方案必须充分考虑现有技术栈、团队能力、时间成本等现实约束
3. **方案可比原则**：多个候选方案必须在同一维度（可行性、成本、风险）上进行对比
4. **步骤可执行原则**：每个实施步骤必须具体到"谁做什么、产出什么、如何验证"
5. **渐进落地原则**：优先拆分为可独立交付的阶段，每阶段有明确的交付物和验收标准

### 步骤 1：需求理解与目标对齐
- **原始需求梳理**：明确用户提出优化的背景和动机
- **业务目标提取**：从需求中提炼出可衡量的业务目标（如性能提升 X%、开发周期缩短 Y 天）
- **验收标准定义**：与用户对齐"优化完成"的具体标准
- **约束条件收集**：明确不可变的技术栈、团队规模、上线时间等硬性约束

**输出**：需求理解摘要（口头确认，不单独成文）

---

### 步骤 2：现状分析
- **现有方案梳理**：分析当前技术方案的架构、核心模块、数据流
- **痛点识别**：定位现有方案中模糊、缺失、过于抽象的部分
- **技术约束盘点**：
  - 已有技术栈和依赖
  - 已有代码结构和约定
  - 团队熟悉的技术领域
- **项目上下文**：
  - 当前项目阶段（新建/迭代/维护）
  - 已有文档和设计资产
  - 历史决策和约束

**输出**：现状分析结论（纳入最终文档）

---

### 步骤 3：方案对比与决策
- **候选方案设计**：针对优化目标，提出 2-3 个可行方案
- **对比维度**（统一标准）：

| 维度 | 说明 |
|------|------|
| 可行性 | 技术上是否可实现，团队是否有能力 |
| 成本 | 开发工作量、基础设施成本、维护成本 |
| 风险 | 技术风险、进度风险、兼容性风险 |
| 收益 | 对业务目标的达成程度 |
| 扩展性 | 未来演进的灵活性 |

- **推荐标注**：在对比矩阵中标注推荐方案，说明推荐理由
- **用户决策**：如有多个可行方案，提供给用户选择

**输出**：方案对比矩阵（纳入最终文档）

---

### 步骤 4：优化方案详细设计
**目标**：将推荐方案细化到可直接指导实施的程度。

1. **整体架构设计**
   - 优化后的架构图（文字描述或 ASCII 图示）
   - 与现有架构的差异说明
   - 新增/变更的模块和职责

2. **核心模块设计**
   - 每个模块的输入/输出/职责
   - 模块间的接口契约
   - 关键数据结构定义

3. **接口定义**（如涉及）
   - API 入参/出参/异常
   - 事件/消息格式
   - 第三方集成点

4. **数据结构设计**（如涉及）

### 步骤 5：分步实施计划
**目标**：将详细设计拆分为可独立交付的实施阶段。

**拆分原则**：
- 每阶段可独立编译/运行/验证
- 阶段间有明确的依赖关系和交付物
- 每阶段预计工作量不超过 2-3 天

**阶段结构**：
```markdown
### 阶段 N：{主题}（预计 X 天）

**目标**：本阶段要达成的具体目标

**前置依赖**：依赖哪个阶段的哪些产出

**实施步骤**：
- [ ] 步骤 1：{具体动作} → 产出：{产出物} → 验证：{如何验证}
- [ ] 步骤 2：{具体动作} → 产出：{产出物} → 验证：{如何验证}

**验收标准**：
- [ ] 标准 1
- [ ] 标准 2
```

**输出**：更新 `docs/solution-optimization/{日期}-{主题}.md`，补充分步实施计划
