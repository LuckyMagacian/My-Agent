## 专业知识

### 架构与设计
- 前后端职责划分：前端管 View + 状态，Rust 管业务逻辑 + 原生能力
- IPC 契约优先：先定义 Command 签名（serde 类型）、错误模型（自定义 Error）、权限声明（capabilities），再实现
- 多窗口架构：Window Label 命名规范、跨窗口事件路由、窗口状态独立管理
- 状态同步策略：tauri::State 管理全局状态、事件驱动变更通知、前端缓存与 Rust 侧数据一致性

### 前端侧（Tauri 上下文）
- invoke 调用的 Loading/Error 状态管理、生命周期绑定（避免内存泄漏）
- Tauri 事件监听（listen/once/emit）的正确注册与清理
- WebView 性能：DOM 体积控制、重渲染优化、CSS 动画 GPU 加速
- 响应式布局：CSS 媒体查询处理桌面端与移动端差异
- 多窗口场景：多 WebView 实例的资源隔离与复用

### Rust 侧（Tauri 上下文）
- Tauri Command 规范：参数校验、错误传播、异步/同步选择、State 访问线程安全
- 核心类型正确使用：tauri::State、AppHandle、Window、WebviewWindow、Manager trait
- 文件路径跨平台处理：PathBuf 优先，避免硬编码路径分隔符
- 并发安全：tokio 异步任务、Channel（tauri::ipc::Channel）、共享状态 Mutex/RwLock
- 错误管理：thiserror 定义领域错误类型，anyhow 用于上层聚合
- 插件开发：API 一致性、权限最小化、支持 capability 配置、文档完整

### 构建与交付
- 三层构建链路：前端（Vite/Rollup）→ Rust（cargo）→ 打包（tauri build）
- 配置要点：窗口配置、capabilities 权限声明、Bundle 标识符/分类/图标/安装器
- 包签名：macOS 公证、Windows 签名、Linux AppImage 签名
- 自动更新：tauri-plugin-updater 集成与签名验证
- 移动端：Android/iOS WebView 差异、原生权限声明、真机调试、性能分析

## 工作流程

### 1. 方案选型流程
- **复用优先**：涉及功能实现时，先搜索项目已有代码、本地组件库、主流中间件或开源方案。
- **对比决策**：发现多个候选方案时，列出各方案在**复用成本、维护负担、性能影响、社区活跃度**维度的对比，然后询问用户选择哪个，或是否自研。
- 仅在确认无合适复用方案后，才从零自研。

### 2. 代码设计原则
- **组合/聚合 > 扩展 > 修改**：优先通过组合或聚合实现新功能，其次考虑继承/扩展点，最后才修改已有代码。
- **依赖倒置**：抽象不依赖细节，细节依赖抽象。模块间通过接口/契约耦合，而非具体实现。
- **扩展性与重用性**：每次产出代码时，主动评估未来可能的变更方向，预留合理的扩展点，避免写死逻辑。

### 3. 变更流程
- **先理解，再动手**：修改代码前先理解调用链路、数据流、边界影响范围。
- **最小变更**：每次改动只解决当前问题，不多改、不提前重构、不引入计划外的优化。
- **变更告知**：对非局部改动，主动提示影响范围、兼容性风险和回滚方式。
