# 必须遵守的规则

## 交流规范

- 使用简体中文交流，专业术语保留英语，若无法使用简体中文则回退英语
- 简洁明了沟通，不要有过多描述性语句
- 不要读取整个文件，使用 grep 或指定行范围查找

## 核心原则：文档先行，依照文档执行

**工作流程**：先产出文档 → 用户确认文档 → 严格按文档执行

**强制规则**：

1. **文档是执行依据**：任何编码、修改、修复动作，必须有对应文档作为依据
2. **先写后做**：禁止直接写代码，必须先完成需求/设计/计划文档
3. **按文档执行**：执行时必须对照文档内容，不得随意偏离
4. **偏离需确认**：发现文档有误或需调整，必须暂停执行，更新文档并获得用户确认后再继续

**文档路径**：`{项目根目录}/docs/`

| 场景     | 文档路径                             | 作用         |
| -------- | ------------------------------------ | ------------ |
| 需求     | `docs/需求说明书.md`               | 定义做什么   |
| 设计     | `docs/设计说明书.md`               | 定义怎么做   |
| 详细设计 | `docs/design/{模块}.md`            | 定义具体实现 |
| 变更计划 | `docs/changes/{日期}-{主题}.md`    | 定义改什么   |
| 问题修复 | `docs/bugs/{日期}-{问题}.md`       | 定义修什么   |
| 分析报告 | `docs/analysis/{模块}-分析报告.md` | 记录分析结果 |

## 开发流程（必须遵循 `software-development-workflow`）

### 完整需求开发

```
需求文档 → 用户确认 → 设计文档 → 用户确认 → 详细设计 → 用户确认 → 按文档编码
```

### 功能变更

```
变更计划文档 → 用户确认 → 按文档执行变更
```

### 问题修复

```
问题分析文档 → 用户确认 → 按文档执行修复
```

### 技术分析

```
分析过程 → 输出分析报告文档
```

## 质量保障

- **全程使用** `attention-maintenance` 维持任务注意力，每 10 轮检查一次任务锚点
- **执行对照**：执行每一步时，必须确认是否符合文档规定

## 问题修复规则

- 在分析任何代码错误（尤其是框架报错）时，**禁止直接给出猜测**
- 必须遵循 `software-development-workflow` 中的问题修复流程
- 必须产出问题分析文档，经用户确认后再执行修复

## 技术场景 Skills（按需触发）

- React/Next.js 代码：参考 `react-best-practices`
- React Native/Expo 代码：参考 `react-native-skills`
- 页面过渡动画：参考 `react-view-transitions`
- HTTP 请求/网络操作：参考 `http-retry-handler`
- UI/UX 原型设计：参考 `prototype-prompt-generator`
